## 920セッション分を分析したら見えたもの

Claude Codeは、Anthropicが提供するAIコーディングアシスタントのCLIツールです。ターミナルでチャットしながらコードを書いたり、ファイルを編集したり、gitの操作をしたりできます。

これを半年くらい使い続けて、ある日ふと「自分がどれだけ非効率なことしてるか」気になったんですよね。

セッションログを分析してみたら、11,633エントリ・920セッション分のデータが溜まってました。そこから見えてきた事実がなかなかしんどかったです。

| 非効率パターン | 回数 |
|--------------|------|
| AIからの確認に「はい」「yes」「y」 | 772回 |
| 外部ツール連携の手動初期化 | 234回 |
| `git push`の手動操作 | 212回 |
| 会話が長くなって文脈を忘れた後の「続きやって」 | 97回 |

772回の「はい」って、毎セッションで何回もyes打ってるってことです。これ全部自動化できるんじゃないかと思って、Claude Codeの設定をまるごとGitリポジトリで管理する仕組みを作りました。

https://github.com/DaichiHoshina/ai-tools

半年で236コミット積んで、今こんな構成になってます。

```
ai-tools/claude-code/
├── commands/     30個のカスタムコマンド（チャットで /xxx と打って実行）
├── skills/       22個のスキル定義
├── agents/       10個のエージェント定義
├── guidelines/   44個のガイドライン
├── hooks/        16個のイベントHook（計1,042行）
├── lib/          21個の共通ライブラリ（計3,250行）
├── install.sh    初回インストール
└── sync.sh       設定同期
```

## Hooksが一番面白い

Claude Codeの特定のイベントに反応して自動実行されるシェルスクリプトです。16個作ったんですが、特に効いてるのがこの3つです。

### 1. 危険操作の自動ブロック

`pre-tool-use.sh`で、ツール実行前に3層分類してます。

| 分類 | 例 | 挙動 |
|------|-----|------|
| **Safe** | Read, Grep, git status | そのまま実行 |
| **Boundary** | Edit, Write, git commit | 内容次第で確認 |
| **Forbidden** | `rm -rf /`, `git push --force` | 拒否 |

Bashコマンドは内部でさらに4段階に分類してて、`prettier --fix`みたいな自動整形はBoundary、`ls`みたいな読み取り系はSafeに振り分けてます。Safe操作のときはメッセージ出力なしにして、APIの消費量も抑えてます。

### 2. 完了通知

長いタスクを投げて別の作業をしてることが多いので、タスク完了時にデスクトップ通知を飛ばしてます。`terminal-notifier`でmacOSの通知バナーを出して、クリックするとiTerm2がアクティブになります。外出中は`ntfy.sh`でモバイルにも通知が飛びます。

地味ですが、これがないと「終わってたのに気づかなくて10分放置」が頻発してました。

### 3. 会話の文脈を自動バックアップ

Claude Codeは会話が長くなると、古い部分を自動で圧縮（compact）して忘れてしまいます。「さっき言ったやつの続きやって」が通じなくなるんですよね。97回も「続きやって」と打ってたのはこれが原因です。

そこで、圧縮が走る直前に作業状態を自動保存して、圧縮後に自動復元するHookを作りました。保存項目は、現在のタスク、編集中のファイル、次のアクション、直前のユーザー発言3件、使用中のコマンド名です。

## 「マージすんなってば！！」事件

振り返りログを見てたら、「マージすんな」と5回以上キレてた記録が残ってました。Claude Codeが勝手にMRをマージしちゃう問題です。

毎回チャットで注意してたんですが、セッションが変わると忘れるんですよね。そこで、CLAUDE.md（プロジェクトルートに置くClaude Code用の設定ファイル）にルールとして明記しました。

```markdown
## Git マージ禁止ルール
MR/PRのマージは**絶対に自動実行しない**。
ユーザーが明示的に「マージして」と言った場合のみ実行可。
```

同じように「ダークモード不要」の繰り返し指示が125件あったので、UIデフォルト設定としてルール化しました。

**口で言うんじゃなくて、コードで書く。** これがClaude Codeとの付き合い方で一番大事な教訓でした。

## 自然言語トリガー

212回の手動pushを見て、pushコマンドの歴史を振り返ったらこうなってました。

```
手動 git commit + git push（212回の手動操作）
  ↓
/commit-push-main, /commit-push-pr, /branch-push-mr（3コマンド並立期）
  ↓
/git-push に統合（--main, --pr, --branch オプション）
  ↓
自然言語トリガー追加（現在）
```

今は「pushして」と打つだけで`/git-push`が走ります。日本語の表記揺れも全部カバーしてます。

| 入力 | 実行されるコマンド |
|------|-----------------|
| 「pushして」 | `/git-push`（ブランチ自動判定） |
| 「mainにpush」 | `/git-push --main` |
| 「レビューして直してpush」 | `/review-fix-push` |
| 「おまかせ」 | `/flow --auto`（完全自律モード） |

## 週1の振り返りで数字を追う

`/retrospective`コマンドで、週1回セッション履歴を分析するようにしました。半年で6回実施してます。

| 指標 | 初回（2月） | 最新（3月） | 改善率 |
|------|-----------|-----------|--------|
| 確認応答（yes/y/はい）比率 | 6.6% | 3.9% | **-41%** |
| 外部ツールの手動初期化 | 234回 | 1回 | **-99.6%** |
| 平均メッセージ/セッション | — | 6.1 | — |

改善のサイクルはこうです。

```
/retrospective（週1）→ 問題パターン特定 → 改善案生成 → 実装・反映 → 次回で効果測定
```

改善提案にはBefore/After数値を必ず含めて、次回のretrospectiveで定量検証してます。「なんとなく良くなった」じゃなくて、数字で確認するのがポイントです。

## 日常のコマンドの使い分け

1日に150〜300メッセージくらい送ってます。全期間の使用回数で見ると、使い分けがはっきりしてます。

| やりたいこと | コマンド | 使用回数 |
|------------|---------|---------|
| タスクを丸投げ | `/flow` | 403回 |
| コードレビュー | `/review` | 181回 |
| レビュー→修正→push | `/review-fix-push` | 154回 |
| やることが明確な実装 | `/dev` | 137回 |

`/flow`は内部で複数のAIエージェントが役割分担して動きます。要件分析→タスク分割→実装を自動で進めてくれて、タスクの内容も自動判定します。バグ修正なら要件整理をスキップして直接実装に入る、みたいな判断も勝手にやってくれます。

`/review-fix-push`が3番目に多いのは、毎日の作業完了時に「レビューして直してpush」って打つのが習慣になってるからですね。

## セットアップ

```bash
git clone https://github.com/DaichiHoshina/ai-tools.git ~/ai-tools
cd ~/ai-tools && ./claude-code/install.sh
```

これで`~/.claude/`配下に全部リンクされます。設定を変更したら`sync.sh`で双方向同期できます。

## おわりに

Claude Codeのカスタマイズって、最初は「便利なコマンド作ろう」くらいの気持ちで始めたんですが、半年やってみて分かったのは**設定を育てる**感覚が大事だということです。

772回の「はい」も、5回の「マージすんな」も、212回の手動pushも、全部「繰り返してる非効率」を見つけて自動化していった結果が今の構成です。最初から30コマンド・22スキル・16Hookを作ったわけじゃなくて、振り返りで問題を見つけるたびに1つずつ増やしていきました。

まずは`/flow`、`/dev`、`/review`の3つだけで始めてみてください。使っていくうちに「ここ自動化したいな」と思ったらHookを追加して、「このルール毎回言ってるな」と思ったらCLAUDE.mdに書く。その繰り返しです！
