## 30個作って、毎日使うのは3つ

Claude Codeを1年くらい使い続けてます。カスタムコマンドも気づけば30個になったんですが、正直に言うと毎日使ってるのは3つだけです。

全期間の使用回数を集計してみました。

| コマンド | 使用回数 | ひとことで言うと |
|---------|---------|----------------|
| `/flow` | 403回 | タスクを丸投げ |
| `/review` | 181回 | コードレビュー |
| `/review-fix-push` | 154回 | レビュー→修正→pushを一括 |
| `/dev` | 137回 | 直接実装 |

上位3つで全体の8割くらい占めてます。この記事では、実際に毎日使ってるコマンドを「こう打つとこう動く」の具体例付きで紹介します。

ちなみにカスタムコマンドは `~/.claude/commands/` にMarkdownファイルを置くだけで作れます。中身はプロンプトのテンプレートみたいなものです。

## 迷ったら /flow

一番使ってるコマンドです。何をやるか伝えるだけで、あとは勝手にやってくれます。

```
/flow ユーザー認証機能を追加して
```

これだけで、内部でタスクの種類を自動判定して、適切なワークフローが走ります。

| こう打つと | こう判定されて | こう動く |
|-----------|--------------|---------|
| 「認証機能を追加」 | 新機能実装 | 要件整理→設計→実装→テスト→レビュー→PR |
| 「このバグを修正」 | バグ修正 | 原因特定→実装→テスト→PR |
| 「緊急、本番障害」 | 緊急対応 | 原因特定→修正→テスト→mainに直push |
| 「リファクタして」 | リファクタリング | 設計→実装→テスト→レビュー→PR |
| 「テスト追加して」 | テスト作成 | テスト作成→レビュー→テスト実行→PR |

全部で11種類のタスクタイプがあります。バグ修正なら要件整理をスキップして直接原因特定に入るし、緊急対応ならPR作成じゃなくてmainに直pushする、みたいな判断を自動でやってくれます。

### 完全自律モード

```
/flow --auto ユーザー一覧のページネーションを実装
```

`--auto`をつけると、途中の確認を全部スキップして最後まで自動で走ります。レビューでCriticalが出たら自動修正して、問題なくなるまで最大3回ループします。終わったらpushまでやってくれるので、投げたら放置でOKです。

日本語で「おまかせ」って打っても同じ動きをします。

## やること決まってるなら /dev

`/flow`はワークフロー全体を回すので、ちょっとした修正にはオーバーです。「ここをこう直して」が明確なときは`/dev`を使います。

```
/dev このバリデーションメッセージを日本語に変更して
```

`/flow`との違いは、エージェント階層（要件分析→タスク分割→実装の多段構成）を使わずに直接実装に入るところです。余計なステップがないぶん速いです。

### --quickで爆速

```
/dev --quick READMEのtypoを修正
```

`--quick`をつけると、軽量モデル（Haiku）で実行されます。1〜2ファイルの軽微な修正ならこれで十分です。

使い分けの目安はこんな感じです。

| 状況 | コマンド |
|------|---------|
| 何をやるかざっくり伝えたい | `/flow` |
| 修正内容が具体的 | `/dev` |
| 1-2ファイルの軽微な修正 | `/dev --quick` |
| 完全に任せて放置したい | `/flow --auto` |

## 変更したら /review

コードを変更したら、pushする前にこれを通してます。

```
/review
```

引数なしで打つと、`git diff`の差分に対して7観点のレビューが走ります。

| 観点 | チェック内容 |
|------|------------|
| 設計 | クリーンアーキテクチャ、依存方向 |
| 品質 | 型安全性、パフォーマンス |
| 可読性 | 命名、認知的複雑度 |
| セキュリティ | OWASP Top 10、エラーハンドリング |
| ドキュメント/テスト | カバレッジ |
| 恒久対応 | 対症療法になってないか |
| ログ | レベル適切性、機密情報の混入 |

結果はCritical/Warning/Infoに分類されます。Criticalは必ず直す、Warningは判断、Infoは参考、という運用です。

### MR/PRのレビューもできる

```
/review https://github.com/xxx/xxx/pull/123
```

URLを渡すとリモートのPR差分を取得してレビューしてくれます。PR番号だけでもOKです。セルフレビューにも、他メンバーのPRの事前確認にも使えます。

## 作業終わりの定番: /review-fix-push

使用回数154回。一番定着したコマンドです。毎日の作業完了時にこれを打つのが習慣になってます。

```
/review-fix-push
```

やってることは名前の通りです。

1. 7観点のレビューを実行
2. CriticalとWarningを自動修正
3. コミットしてmainにpush

以前は`/review` → 指摘を見て手動修正 → `/commit` → `git push`と何ステップも踏んでたのが、1回で済むようになりました。「レビューして直してpush」って日本語で打っても同じです。

## Git操作: /git-push

```bash
/git-push --main            # mainにpush
/git-push --pr              # PR作成
/git-push --branch feat/auth  # ブランチ作成→push→PR
```

`git add` → `git commit`（メッセージ自動生成）→ `git push`を1コマンドでやってくれます。コミットメッセージはdiffを分析して自動生成されるので、自分で考える必要がありません。

日本語でも打てます。

| 入力 | 動作 |
|------|------|
| 「pushして」 | ブランチ自動判定でpush |
| 「mainにpush」 | mainに直push |
| 「ブランチ切って」 | 新規ブランチ作成→push→PR |

## バグに遭遇したら /diagnose

エラーログをそのまま貼って打つだけです。

```
/diagnose
panic: runtime error: index out of range [3] with length 2

goroutine 1 [running]:
main.processItems(...)
    /app/handler.go:45
```

ログを解析して、原因の特定から修正提案までやってくれます。スタックトレースがあれば該当コードまで読みに行くので、見知らぬコードベースのバグ調査で特に助かってます。

## 設計を考えるとき: /plan

```
/plan 注文キャンセル機能の設計
```

実装はせずに、設計だけ出力してくれます。ディレクトリ構造、インターフェース設計、実装フェーズの分割まで提案してくれるので、「まず設計固めたい」ときに使います。実装に移りたくなったら`/dev`で始めればOKです。

## コードベースの把握: /explore

新しいプロジェクトに入ったときや、全体像を知りたいときに使います。

```
/explore このプロジェクトの認証フローを教えて
```

内部で最大4つの探索エージェントが並列で動いて、異なる観点から同時に調査してくれます。1つがミドルウェアを、別の1つがユーザーモデルを調べる、みたいな分担です。

## おわりに

30個もコマンドを作っておいてアレですが、結局のところ日常で回してるのは `/flow`、`/dev`、`/review` の3つです。

これから始める人は、この3つだけで十分です。カスタムコマンドは `~/.claude/commands/` にMarkdownを置くだけなので、「この操作いつもやってるな」と思ったときに作ればいいと思います。
